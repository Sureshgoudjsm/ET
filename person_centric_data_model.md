# Person-Centric Data Model and Workflow

To effectively track proxy payments and informal loans, the existing data model needs to be enhanced to prioritize the relationship between the user and other individuals. This involves creating a robust `Person` entity and linking all relevant financial transactions to these individuals.

## 1. Enhanced Data Model: The `Person` Entity

The core of this new approach is a more comprehensive `Person` entity. Instead of just a name, each person should have attributes that facilitate detailed tracking.

### `Person` Entity Attributes:

| Attribute Name      | Data Type    | Description                                                                                             | Example Value                                |
| :------------------ | :----------- | :------------------------------------------------------------------------------------------------------ | :------------------------------------------- |
| `person_id`         | UUID         | Unique identifier for each person.                                                                      | `uuid-1234-abcd`                             |
| `name`              | String       | Full name or nickname of the person.                                                                    | `Brother (Sunny)`                            |
| `relationship_type` | Enum         | Defines the relationship to the user (e.g., `Family`, `Friend`, `Other`).                               | `Family`                                     |
| `contact_info`      | String       | Optional: Phone number or email for reminders.                                                          | `+91-9876543210`                             |
| `notes`             | Text         | Any additional notes about the person.                                                                  | `Brother-in-law, lives in Mumbai`            |
| `total_owed_to_user`| Decimal      | Calculated: Sum of all active debts this person owes the user.                                          | `₹56,000`                                    |
| `total_user_owes`   | Decimal      | Calculated: Sum of all active debts the user owes this person.                                          | `₹0`                                         |

## 2. Workflow for Proxy Payments and Informal Loans

This section outlines the proposed workflows for managing different types of person-centric financial interactions.

### 2.1. Proxy Payments (EMIs, Gold Loan Interest, Other Loans)

**Scenario:** The user pays an EMI or interest on behalf of another person.

**Workflow:**
1.  **Initiate Payment Record:** The user navigates to a new "Proxy Payments" section or directly from the `Person` profile.
2.  **Select Payer (User) and Payee (Bank/Lender):** Standard transaction fields.
3.  **Specify `On Behalf Of` Person:** The user selects the `Person` (e.g., Brother, Cousin) for whom the payment is being made.
4.  **Enter Amount and Date:** The actual amount paid and the date of payment.
5.  **Categorize Payment:** The user categorizes the payment (e.g., `EMI - Brother's Loan`, `Gold Loan Interest - Cousin`). This helps in personal expense tracking.
6.  **System Action:**
    *   Records the payment as an expense for the user.
    *   Automatically creates a corresponding `Debt` entry for the `On Behalf Of` person to the user. This `Debt` entry will be linked to the original payment transaction.
    *   Updates `Person.total_owed_to_user` for the `On Behalf Of` person.
7.  **Repayment Tracking:** When the `On Behalf Of` person repays the user, the user records a `Repayment` transaction, which reduces the `Debt` entry and updates `Person.total_owed_to_user`.

**Example Flow (EMI for Brother):**
*   User pays `₹20,000` for Brother's EMI.
*   System records: User Expense: `₹20,000` (Category: `EMI - Brother's Loan`).
*   System records: Brother owes User: `₹20,000`.

### 2.2. Informal Short-Term Lending (Lent/Borrowed from Friends/Family)

**Scenario:** The user lends money to a friend or borrows money from a friend.

**Workflow (Lending Money):**
1.  **Initiate Lending Record:** The user navigates to a "Lend/Borrow" section or from the `Person` profile.
2.  **Select `Person`:** The user selects the friend/family member (e.g., Friend A).
3.  **Enter Amount Lent:** The amount provided to the person (e.g., `₹1,000`).
4.  **Specify Expected Repayment Date:** (e.g., `10 days from now`).
5.  **System Action:**
    *   Records an outflow transaction for the user (Category: `Lent to Friend A`).
    *   Creates a `Debt` entry for Friend A to the user, with the amount and expected repayment date.
    *   Updates `Person.total_owed_to_user` for Friend A.
    *   Sets up an optional reminder for the repayment date.
6.  **Repayment:** When Friend A repays, the user records a `Repayment` transaction, clearing the `Debt` entry and updating `Person.total_owed_to_user`.

**Workflow (Borrowing Money):**
1.  **Initiate Borrowing Record:** The user navigates to a "Lend/Borrow" section or from the `Person` profile.
2.  **Select `Person`:** The user selects the friend/family member (e.g., Friend B).
3.  **Enter Amount Borrowed:** The amount received from the person (e.g., `₹10,000`).
4.  **Specify Expected Repayment Date:** (e.g., `1 month from now`).
5.  **System Action:**
    *   Records an inflow transaction for the user (Category: `Borrowed from Friend B`).
    *   Creates a `Debt` entry for the user to Friend B, with the amount and expected repayment date.
    *   Updates `Person.total_user_owes` for Friend B.
    *   Sets up an optional reminder for the repayment date.
6.  **Repayment:** When the user repays Friend B, the user records a `Repayment` transaction, clearing the `Debt` entry and updating `Person.total_user_owes`.

## 3. Consolidated Debt Tracking

A central "People & Debts" dashboard would be crucial, showing:
*   **Summary by Person:** For each `Person`, display `total_owed_to_user` and `total_user_owes`.
*   **Detailed Debt List:** A list of all active `Debt` entries, including who owes whom, amount, original transaction, and expected repayment date.
*   **Reminders:** Automated notifications for upcoming repayment dates for both lent and borrowed amounts.

This person-centric model allows for a clear separation of the user's personal expenses from the financial obligations they manage on behalf of others, while still providing a comprehensive view of all financial relationships.
